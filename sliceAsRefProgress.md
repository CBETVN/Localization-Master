# Slice-as-Reference SO Scaling

## Goal

After a Smart Object is translated, its text content may become larger (longer words, different language metrics). The SO's visual size can then exceed its intended export area.

The idea: use the **Photoshop document slice** that spatially overlaps the SO as the container reference. If the SO's bounds are larger than the overlapping slice, scale the SO down uniformly so it fits inside.

This avoids text overflow in exported assets without needing paragraph text, auto-fit text engines, or manual resize per language.

---

## How it fits into the existing plugin flow

The scale step runs **after** `translateSmartObject` completes, inside STEP 8 of `processMatchedFolder` in `parsingLogic.js`. It is non-fatal — if no slice is found or scaling fails, the translation run continues normally.

```
translateSmartObject(child.layer, text)   ← already works
    ↓
fitSOToOverlappingSlice(child.layer)      ← new, optional post-step
    ↓
processedIds.add(soId)                    ← dedup, unchanged
```

## Slice data — how it is read

Confirmed via SliceDirector codebase (`SliceDirector/src/index.jsx → getUserGeneratedSlices`).

The descriptor path is:

```js
batchPlay([{ _obj: "get", _target: [{ _ref: "document", _enum: "ordinal", _value: "targetEnum" }] }])
  → result[0].slices.slices   // array of all slices
```

Each slice object has:
- `sliceID` — unique identifier
- `name` — user-given name
- `bounds.top/left/bottom/right` — pixel coordinates in document space
- `origin._value === "autoGenerated"` — filter flag; auto-slices are excluded

---

## Matching logic — SO to slice

Both the SO and the slices exist in the same document coordinate space, so overlap is a simple axis-aligned rectangle test:

```
overlap_left   = max(so.left,   slice.left)
overlap_top    = max(so.top,    slice.top)
overlap_right  = min(so.right,  slice.right)
overlap_bottom = min(so.bottom, slice.bottom)

overlap_area = (overlap_right - overlap_left) * (overlap_bottom - overlap_top)
               — zero or negative means no overlap
```

If the SO overlaps multiple slices, the one with the **largest overlap area** wins. This is the most stable heuristic when SOs sit near slice boundaries.

---

## Scale calculation

$$
s_{raw} = \min\left(\frac{W_{slice}}{W_{so}},\ \frac{H_{slice}}{H_{so}},\ 1\right)
$$

- Uniform scale (width = height) to preserve aspect ratio.
- Capped at `1` — never upscales.
- Clamped at `minScale` floor (default `0.65`) — safety net for badly-matched slices.

The transform uses `QCSAverage` (center of current bounds) as the anchor, so the SO stays centered in place after scaling.

## Proposed new code in `photoshop.js`

Two new exported functions to add at the bottom of the file:

### `getUserGeneratedSlices()`

Reads document slices via batchPlay, filters out auto-generated ones, returns flat array of `{ id, name, top, left, bottom, right }`.

Must be called inside `executeAsModal`.

### `fitSOToOverlappingSlice(soLayer, minScale = 0.65)`

1. Reads SO bounds from `soLayer.bounds` (no extra batchPlay).
2. Calls `getUserGeneratedSlices()`.
3. Finds best-overlap slice.
4. Computes uniform scale.
5. If `rawScale >= 1` → returns `{ ok: true, scaled: false, reason: "already_fits" }`.
6. Applies `transform` batchPlay action with `percentUnit` width/height.
7. Returns `{ ok, scaled, scalePercent, sliceName }` or `{ ok: false, reason }`.

Must be called inside `executeAsModal`.

## Known risks / open questions

- `soLayer.bounds` on a Smart Object may include layer effects (drop shadows etc.) which inflate the visible bounds beyond the actual content. If that causes over-shrinking, the fix is to use batchPlay `get` to read `boundsNoEffects` instead.
- The `minScale = 0.65` floor is a guess. It may need tuning per project.
- `QCSAverage` centers the scale on the layer's current bounding box. If the SO is not centered inside its slice, this may shift it off-center after scaling. Alternative: compute the slice center and anchor the transform there.
- Consider a detection-only dry run pass first (log overlap + computed scale, no transforms) before enabling actual scaling.
